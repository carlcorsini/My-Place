Aggregation with SQL

Related Standard: SQL0010-V1 - Write queries that filter and aggregate data using SQL

Objectives

  • Write aggregation queries with COUNT, MAX, MIN, SUM, AVG, etc.
  • Explain the role of GROUP BY in aggregate queries

Rationale

In a relational database, data of the same type is stored in the same logical entity or table. When developing an application, generally each persisted entity will have its own database table but may have relations to other tables by the use of columns which reference another entity, generally by that entity's id.

Many critical business-related questions are not focused on the details of a specific entity, but instead looking across all (or some subset of) entities of the same type to identify trends or derive metrics for performance (as a company, application, etc).

For example, suppose we are building an ecommerce application. In this case we might consider having a table named transactions which tells us about purchases our customers make. We might store what customer bought something, when they bought it, and how much they spent (subtotal, tax, and total). While this type of storage is useful for developers, it is also useful for building queries to answer questions the business people will ask about the performance of the company.

Many business questions include the word "by" or can be translated into an equivalent statement using "by". Presence of "by" in a business question tends to indicate that some type of aggregate query is required. For example:

  • How many transactions do we have by week? By month? By quarter? By year?
  • How much revenue have we made by week? By month? By quarter? By year?
  • How much revenue have we earned by customer by month? By quarter? By year?

Each of the above questions asks us a question not about a singular transaction, but instead about statistics across all transactions, grouped by some criteria. Aggregate queries allow us to solve problems like these; when we have many, granular data points, we can "collapse" them by grouping and using an aggregate function.

The remainder of this lesson will use a table called movies which records the id, title, duration, rating, genre, is_3d, released_at, score, summary. Write aggregation queries with COUNT, MAX, MIN, SUM, AVG, etc.

Movie Corp is a scrappy startup, so the CEO asks us, the only developer, to give a weekly update on how many movies we currently have.

To count the number of rows in a given table we use the COUNT aggregate function:

-- PostgreSQL & MySQL
SELECT COUNT(*)
FROM movies;

You should see something like this given our sample data set from above:

root@localhost [movie_junkies_dev]> SELECT COUNT(*)
    -> FROM movies;
+----------+
| COUNT(*) |
+----------+
|        8 |
+----------+
1 row in set (0.00 sec)

So, we proudly tell the CEO, "there are 8 movies right now."

Next, we decide out of curiosity to ask, how long is the longest movie? For this we can get the maximum value of duration column in the movies table using the MAX aggregate function:

-- PostgreSQL & MySQL
SELECT MAX(duration)
FROM movies;

We see something like this:

+---------------+
| MAX(duration) |
+---------------+
|         51420 |
+---------------+
1 row in set (0.00 sec)

Like MAX and COUNT, MIN, SUM, and AVG will return the respective computation over values in the rows. Here are some examples to try:

-- PostgreSQL & MySQL
SELECT MIN(duration)
FROM movies;
-- PostgreSQL & MySQL
SELECT SUM(duration)
FROM movies;
-- PostgreSQL & MySQL
SELECT AVG(duration)
FROM movies;

In summary, the general form for using an aggregate function over the entire range of rows is:

SELECT AGGREGATE_FUNCTION(column_name)
FROM table;

Explain the role of GROUP BY in aggregate queries

So far we applied an aggregate function across every row of the table, that is, we are performing the aggregate function over the entire data set. Suppose we wanted to ask a more detailed question; how many movies are 3D and how many are not 3D?

The GROUP BY clause sets a condition on which to match rows before applying an aggregate function. Another way to think of this is that a column, c, contains entries for many rows, r1, r2, ... rn,. The distinct values of the rows for a given column determine the values that will be grouped on.

Once the rows are matched on the GROUP BY clause then data contained in the specified column(s) for each group is provided to the aggregate function specified in the SELECT clause. This can best be illustrated by an example:

SELECT is_3d,
  count(id)
FROM movies
GROUP BY is_3d;

In this case we first want to group rows on the distinct values of is_3d in the table. In our case is_3d is of type BOOLEAN so there are only two possible values; TRUE and FALSE. Running this query with MySQL we get:

+-------+-----------+
| is_3d | count(id) |
+-------+-----------+
|     0 |         6 |
|     1 |         2 |
+-------+-----------+
2 rows in set (0.00 sec)

In MySQL TRUE is represented by 1 and FALSE is 0, PostgreSQL uses t for TRUE and f for FALSE.

Note that everything that is not provided to an aggregate function is provided to the GROUP BY clause. Another example may make this more clear.

Suppose now we want to know "by rating, how many movies are in 3D vs not in 3D?"

SELECT rating,
  is_3d,
  COUNT(*)
FROM movies
GROUP BY rating, is_3d;

One way to think of the GROUP BY clause in this query is to say "for every value of rating for every value of is_3d show me the count of movies"

Running this query we see:

+--------+-------+----------+
| rating | is_3d | COUNT(*) |
+--------+-------+----------+
| PG     |     0 |        1 |
| PG     |     1 |        1 |
| PG-13  |     0 |        3 |
| PG-13  |     1 |        1 |
| R      |     0 |        2 |
+--------+-------+----------+
5 rows in set (0.00 sec)

Suppose we wanted to ask a slightly different business question, what is the average score for movies by genre and rating?

Deconstructing the ask we know that there is an aggregate value - the average - and two factors by which we which to group the data - genre and rating. Our SQL query corresponds to the ask directly:

SELECT genre,
  rating,
  AVG(score)
FROM movies
GROUP BY genre,
  rating;

which returns:

+-------------+--------+------------+
| genre       | rating | AVG(score) |
+-------------+--------+------------+
| Action      | PG-13  |    7.75000 |
| Adventure   | PG     |    8.10000 |
| Animation   | PG     |    7.60000 |
| Animation   | PG-13  |    8.30000 |
| Comedy      | PG-13  |    7.50000 |
| Crime       | R      |    8.90000 |
| Documentary | R      |    1.20000 |
+-------------+--------+------------+
7 rows in set (0.00 sec)

Finishing up with GROUP BY - the HAVING clause

In review, aggregate functions are used to compute against a "returned column of numeric data" from your SELECT statement. They summarize the results of a particular column of selected data. Although they are required for fields not listed in the "GROUP BY" clause, these functions can be used without the "GROUP BY" clause. For example:

-- PostgreSQL & MySQL
SELECT SUM(duration)
FROM movies;

The HAVING clause allows you to specify conditions on the rows for each group. The HAVING clause is analogous to a WHERE clause.

In other words, the conditions you specify decide if a row's value is given to the aggregate function or not. The HAVING clause should follow the GROUP BY clause.

Consider the example of trying to answer the question, "what is the average score for movies with a rating of PG?" This can be answered with the following query:

-- PostgreSQL & MySQL
SELECT rating,
  AVG(score)
FROM movies
GROUP BY rating
HAVING rating = 'PG';

The HAVING clause is more powerful than just the previous example though, HAVING can be used to filter on the value of an aggregate, for example:

-- PostgreSQL & MySQL
SELECT rating,
  AVG(score)
FROM movies
GROUP BY rating
HAVING AVG(score) > 7.5;

which returns:

+--------+------------+
| rating | AVG(score) |
+--------+------------+
| PG     |    7.85000 |
| PG-13  |    7.82500 |
+--------+------------+
2 rows in set (0.00 sec)
